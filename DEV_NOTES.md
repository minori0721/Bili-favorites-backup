# Dev 测试说明

本文只记录 `dev` 分支相对 `main` 的当前测试内容。确认稳定并合并到 `main` 时，把需要公开给用户的内容整理进 `CHANGELOG.md`，然后重置本文件。

## 当前状态

- 分支：`dev`
- 基准版本：`2.4.0`
- 当前 dev 变更：在既有远端冲突归档和上传节流修复上，增加充电视频识别、七日权限复查、多账号自动接管、缓存统计纠正与临时目录清理修复。
- 镜像预期：推送 `dev` 分支后发布 `minori0721/bili-favorites-backup:dev`；`latest` 仍只由 `main` 分支发布。

## 当前 dev 变更

- 正式上传前先预检整组白名单文件；任一文件同名异大小时，将该组正式路径中已存在的旧文件统一 MOVE 到稳定的 `_history/<UTC时间>/`，再上传当前本地成品。
- 归档阶段在任何 PUT 前完成；MOVE 失败立即停止，不覆盖远端旧文件，也不删除本地缓存。归档完成后即使 PUT 中断，使用同一任务重试也不会重复归档。
- SQLite 收藏夹关系记录最近 20 次远端冲突归档，并清除已经移动的正式路径证明；当前版本全部确认后再写入新的 `remote_files`。
- 正式上传任务重试耗尽后保留为 `failed`，后续扫描重新入队时复用并重置该任务，不再直接删除任务记录。
- 已完成持久队列初始化的数据库在启动时会主动找回“`upload_failed` + 有效本地下载会话 + 无可运行上传任务”的记录，每次最多按 `startupRecoveryBatchSize` 恢复。
- 新增全局 `uploadFileIntervalSeconds`，默认 10 秒、范围 0–120 秒；所有上传任务共享同一 PUT 启动闸门，预检命中和同名同大小跳过不占用间隔，上传流在启动后仍可并发。
- 同一上传任务已有文件完成或预检确认后，后续文件返回 405 会标记为临时上传会话异常，持久任务按 5、10、30 分钟退避；首文件直接 405 继续按确定性错误隔离，不改变现有上传熔断规则。
- 视频详情明确返回`is_upower_exclusive=true`且`is_upower_play=false`时，不再启动BBDown或写入普通失败记录，而是标记为`charging_restricted`并显示“充电视频”。试看字段只用于说明，不作为完整权限。
- 状态兼容schema升级到12，限制信息继续保存在视频`payload_json`；持久队列新增按BVID去重的`access_probe`。SQLite数据库schema升级到2只为重建状态汇总视图，表结构和现有数据不变。
- 首次识别后立即检查其他启用账号；任一账号具备完整权限便用该账号恢复下载并服务全部收藏夹目标。全部无权限时按`7天 ± 12小时`复查，字段未知或临时错误按`6小时 ± 30分钟`复查，无启用账号时24小时后复查。
- 充电探测固定单并发，不占下载并发、不受缓存背压或上传熔断影响；账号扫码刷新、重新启用或新增时会立即唤醒等待中的探测。
- 已确认备份不会因当前充电限制降级；画质重调只暂停下载阶段，其他账号获得权限后同步切换下载账号并继续原持久任务。
- 会话恢复统计只把已验证成品和真实aria2控制文件计为可续传；`_invalid`、`_incompatible`和无法接管的残片计入待清理。字段晚到时只删除本轮新生成且验证失败的试看残片，升级前已有文件不自动删除。
- 清理“全部临时下载文件”时只删除`temp`挂载目录内的子项，不再尝试删除Docker绑定挂载点本身；残片统计与删除共用同一分类，可清除有效会话中的`_invalid/_incompatible`而保留成品和aria2续传轨道。两项同时选择时不会重复执行，前端会显示包含关系和每项具体错误。

## 测试重点

- 远端组内同时存在“同大小旧成品”和“异大小旧成品”时，应把整组旧文件归档，而不是只替换首个冲突文件。
- MOVE 部分失败时不得发送 PUT；修复后重试应继续归档剩余文件。
- 归档完成后首个 PUT 失败时，再次执行不得重复移动已经归档的旧文件。
- 持久上传任务耗尽后应保留为 `failed`，同去重键再次入队应原地恢复为 `pending` 并清零尝试次数。
- `persistent_jobs_bootstrap_v1=complete` 且上传任务缺失时，重启应重新生成待补传任务和稳定归档段。
- 三个并发上传等待者的 PUT 起点应按全局间隔依次放行，远端预检跳过不得消耗上传间隔。
- 上传中途 405 应进入约 5 分钟的首次持久重试并保持熔断关闭；重启补传时前序文件仅通过预检确认，也应算作已有进度。
- 默认回归：`npm test`、`npm run build`。
- 充电字段矩阵应覆盖普通、无权限试看、无权限无试看、有权限、字段未知和明确下架；标题、提示文案和BBDown错误文本不得作为充电判定依据。
- `access_probe`应覆盖七日、六小时、24小时、重启保持`not_before`和多账号短路；充电视频不得进入四列下载卡片或普通失败统计。
- 失败会话没有有效输出和aria2控制文件时不得显示为可续传；旧`_invalid`应进入清理预览且不会被自动删除。
- 清理测试应覆盖`temp`目录根节点保留、有效会话中的无效隔离文件可删除、有效成品和aria2续传文件不受影响，以及“全部临时文件 + 残片”双选去重。

## 已知问题 / 暂缓项

- 自动归档依赖 AList WebDAV `MOVE`。不支持或拒绝 MOVE 的驱动会保留本地文件并继续显示待补传，不会退回覆盖上传。
- 本轮只实现并模拟验证通用 WebDAV 行为，真实 189CloudPC/AList 的整组 MOVE 与重新上传需要 dev 镜像部署后观察。
- 服务器 AList v3.60.0 的 189CloudPC `family + stream` 场景曾在连续上传后出现 `initMultiUpload` 返回空 `uploadFileId`，WebDAV 层对外映射为 405；本轮通过PUT节流和短退避缓解，未修改 AList 驱动本身。
