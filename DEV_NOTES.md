# Dev 测试说明

本文只记录 `dev` 分支相对 `main` 的当前测试内容。确认稳定并合并到 `main` 时，把需要公开给用户的内容整理进 `CHANGELOG.md`，然后重置本文件。

## 当前状态

- 分支：`dev`
- 基准版本：`2.4.0`
- 当前 dev 变更：在既有可靠性修复基础上，完成 SQLite 日常热路径收敛、Debug 日志轮转，以及同一 BV 同画质向多收藏夹目标共享一次下载。

## 当前 dev 变更

- 正式上传前先预检整组白名单文件；任一文件同名异大小时，将该组正式路径中已存在的旧文件统一 MOVE 到稳定的 `_history/<UTC时间>/`，再上传当前本地成品。
- 归档阶段在任何 PUT 前完成；MOVE 失败立即停止，不覆盖远端旧文件，也不删除本地缓存。归档完成后即使 PUT 中断，使用同一任务重试也不会重复归档。
- SQLite 收藏夹关系记录最近 20 次远端冲突归档，并清除已经移动的正式路径证明；当前版本全部确认后再写入新的 `remote_files`。
- 正式上传任务重试耗尽后保留为 `failed`，后续扫描重新入队时复用并重置该任务，不再直接删除任务记录。
- 已完成持久队列初始化的数据库在启动时会主动找回“`upload_failed` + 有效本地下载会话 + 无可运行上传任务”的记录；内存领取数量由`queuePrefetchLimit`限制。
- 新增全局 `uploadFileIntervalSeconds`，默认 10 秒、范围 0–120 秒；所有上传任务共享同一 PUT 启动闸门，预检命中和同名同大小跳过不占用间隔，上传流在启动后仍可并发。
- 同一上传任务已有文件完成或预检确认后，后续文件返回 405 会标记为临时上传会话异常，持久任务按 5、10、30 分钟退避；首文件直接 405 继续按确定性错误隔离，不改变现有上传熔断规则。
- 视频详情明确返回`is_upower_exclusive=true`且`is_upower_play=false`时，不再启动BBDown或写入普通失败记录，而是标记为`charging_restricted`并显示“充电视频”。试看字段只用于说明，不作为完整权限。
- 状态兼容schema升级到13，旧永久失败通过带`purpose`的`access_probe`一次性重新分类；SQLite数据库schema升级到3，增加收藏关系查询列和组合索引，旧数据库在事务中回填。
- 远端命名元数据随下载会话和`remote_files`保存；多P重命名保留P序号，缺日期/画质/编码时跳过，交换/循环命名整批使用临时名并在失败时回滚。
- 画质预览分为`different/unknown/same/in_progress`；unknown默认不选且后端要求`forceUnknown`。同一BVID同档案使用稳定本地目录并串行复用成品。
- 账号删除会取消依赖B站凭据的下载任务、终止对应运行进程并脱离关系；AList补传不依赖账号，可继续释放本地缓存。
- 迁移schema 3增加轻量/完整模式；完整包包含`temp`，使用ZIP64流式处理、CRC/SHA256及资源上限，未列入校验清单的额外文件会被拒绝；导入目标`temp`非空时返回409，切换失败会回滚原文件。
- 日志管理器统一脱敏并在关闭时flush；旧`/api/state`、`/api/cache/clear`、`/api/remote/list`和旧重命名body已删除。
- 首次识别后立即检查其他启用账号；任一账号具备完整权限便用该账号恢复下载并服务全部收藏夹目标。全部无权限时按`7天 ± 12小时`复查，字段未知或临时错误按`6小时 ± 30分钟`复查，无启用账号时24小时后复查。
- 充电探测固定单并发，不占下载并发、不受缓存背压或上传熔断影响；账号扫码刷新、重新启用或新增时会立即唤醒等待中的探测。
- 已确认备份不会因当前充电限制降级；画质重调只暂停下载阶段，其他账号获得权限后同步切换下载账号并继续原持久任务。
- 会话恢复统计只把已验证成品和真实aria2控制文件计为可续传；`_invalid`、`_incompatible`和无法接管的残片计入待清理。字段晚到时只删除本轮新生成且验证失败的试看残片，升级前已有文件不自动删除。
- 清理“全部临时下载文件”时只删除`temp`挂载目录内的子项，不再尝试删除Docker绑定挂载点本身；残片统计与删除共用同一分类，可清除有效会话中的`_invalid/_incompatible`而保留成品和aria2续传轨道。两项同时选择时不会重复执行，前端会显示包含关系和每项具体错误。
- BBDown的`BBDown.config`改写到`os.tmpdir()/bfb-credentials-*`，目录/文件权限显式为`0700/0600`；启动和关闭清理遗留目录，完整迁移排除新旧凭据目录，旧包完成哈希校验后丢弃凭据内容。
- 普通导出和导入前自动备份共用schema 3清单生成器；自动备份生成后立即预览自检。导入会验证配置、账号、日志、SQLite外键及manifest计数，数据库与JSON文件只有在reload全部成功后才提交，回滚失败时保留`.before-migration-*`副本并明确报错。
- `LogManager`只生成一个脱敏对象供历史存储、事件和SSE共用，加载旧日志时也先清洗；上传、调度、清理和队列摘要统一覆盖Authorization、Cookie、`bili_jct`、CSRF、access/refresh key及URL凭据。
- 远端重命名执行前重新核对路径范围、同目录、源存在和目标冲突；MOVE失败后检查旧/临时/新路径，返回`renamed/rolled_back/stranded/conflict/missing`，唯一实际路径会同步回SQLite。
- 画质重调旧备份DELETE失败不再触发完成回调或本地清理；`quality_cleanup`按60秒起步、20%抖动、最长6小时无限持久重试，上传熔断恢复时间优先。
- 删除账号时完整本地成品直接排队原目标补传；未完成下载保留原收藏夹目标并改用其他启用账号。没有替代账号时任务持久暂停，同UID重新扫码会清除`accountDetachedAt`并唤醒下载、画质和权限探测。
- 新增固定依赖`express-rate-limit@8.5.2`；`/api/login`按IP限制15分钟内5次失败，成功不累计。默认管理员密码、默认会话密钥、非Secure Cookie和Cookie导出启用只产生不含实际值的启动警告。
- 第一阶段继续保留`StateManager`兼容门面，但收藏夹分页、启动恢复规范化、远端确认、失败候选、远端文件预览和队列统计改走 SQLite 查询；视频/关系Lazy缓存分别限制为256条，测试钩子会把运行热路径中的`ownKeys`全量枚举判为失败。
- 启动恢复规范化使用`runtime_recovery_normalization_v2`规则标记和每批500条事务；规则未变化时后续启动直接跳过。新增视频状态、收藏夹状态和失败时间组合索引，上传确认直接查询`remote_files`，不再扫描关系JSON。
- Debug日志改为临时文件写完后原子改名，启动和每次写入后执行单实例轮转；默认14天、200文件、256 MiB，仅处理`data/debug`直属完整`.log`，不跟随符号链接或删除未知文件。三个`BFB_DEBUG_LOG_*`环境变量可覆盖默认值。
- 第二阶段新增稳定`artifactKey`，由BVID、规范化画质、编码、Hi-Res、杜比和命名模板计算；API模式与下载账号不进入成品键，但仍保留在下载会话兼容指纹中。
- `quality_download`去重键改为`quality-download:<BVID>:<artifactKey>`，载荷冻结画质档案并保存多个目标。下载运行中新增目标会事务合并到SQLite，完成时重新读取持久载荷，再为每个目标分别生成上传、替换和清理任务。
- 不同画质档案使用不同本地目录和任务；同一成品只有在对应`artifactKey`的所有目标任务都结束后才允许清理。失败目标保留为持久`failed`任务，因此其他目标成功不会误删共享本地文件。
- 启动时旧的按收藏夹拆分且仍处于下载阶段的画质任务会按成品键合并：目标取并集、尝试次数取最大、`not_before`取最晚；已经进入上传/替换/清理阶段的任务继续独立完成。
- 已完成本地画质成品在账号改派或新目标复用时仍经过`downloadWithBBDown`重新取得当前CID快照；分P变化沿用下载会话的历史归档与缺失分P补齐流程，不直接盲传旧成品。

## 测试重点

- 远端组内同时存在“同大小旧成品”和“异大小旧成品”时，应把整组旧文件归档，而不是只替换首个冲突文件。
- MOVE 部分失败时不得发送 PUT；修复后重试应继续归档剩余文件。
- 归档完成后首个 PUT 失败时，再次执行不得重复移动已经归档的旧文件。
- 持久上传任务耗尽后应保留为 `failed`，同去重键再次入队应原地恢复为 `pending` 并清零尝试次数。
- `persistent_jobs_bootstrap_v1=complete` 且上传任务缺失时，重启应重新生成待补传任务和稳定归档段。
- 三个并发上传等待者的 PUT 起点应按全局间隔依次放行，远端预检跳过不得消耗上传间隔。
- 上传中途 405 应进入约 5 分钟的首次持久重试并保持熔断关闭；重启补传时前序文件仅通过预检确认，也应算作已有进度。
- 默认回归：`npm test`、`npm run build`。
- 充电字段矩阵应覆盖普通、无权限试看、无权限无试看、有权限、字段未知和明确下架；标题、提示文案和BBDown错误文本不得作为充电判定依据。
- `access_probe`应覆盖七日、六小时、24小时、重启保持`not_before`和多账号短路；充电视频不得进入四列下载卡片或普通失败统计。
- 失败会话没有有效输出和aria2控制文件时不得显示为可续传；旧`_invalid`应进入清理预览且不会被自动删除。
- 清理测试应覆盖`temp`目录根节点保留、有效会话中的无效隔离文件可删除、有效成品和aria2续传文件不受影响，以及“全部临时文件 + 残片”双选去重。
- 安全专项应覆盖系统临时凭据权限/清理、完整迁移排除和旧包丢弃；实时日志、历史日志、SSE及队列摘要不得出现分号分隔Cookie或token。
- 迁移专项应覆盖自动备份自检、配置/账号/日志语义错误、SQLite可回滚替换和reload失败后旧数据恢复。
- 重命名专项应覆盖交换、回滚、临时名滞留和多路径冲突；画质清理应覆盖DELETE 404/401/500/超时及无限重试。
- 账号专项应覆盖双账号改派、无替代账号暂停、同UID恢复和完整本地成品直接补传；登录限速应覆盖成功不计数、IP隔离及窗口到期。
- 性能专项应覆盖10,000条视频/关系下日常接口不触发Lazy Proxy全量枚举、缓存不超过256条，以及分页、失败恢复、上传确认和任务领取的查询计划命中索引。
- Debug轮转专项应覆盖年龄、数量、总空间、非法环境变量、原子写入、未知文件保留、符号链接忽略和清理失败不影响下载。
- 共享画质专项应覆盖三个目标只执行一次下载、运行中追加目标、不同档案隔离、旧任务合并、失败目标保留、按成品键清理、清理期间阻止同成品并发下载、耗尽任务可重新提交、账号改派以及完成会话仍重新校验CID。

## 已知问题 / 暂缓项

- 自动归档依赖 AList WebDAV `MOVE`。不支持或拒绝 MOVE 的驱动会保留本地文件并继续显示待补传，不会退回覆盖上传。
- 本轮只实现并模拟验证通用 WebDAV 行为，真实 189CloudPC/AList 的整组 MOVE 与重新上传需要 dev 镜像部署后观察。
- 服务器 AList v3.60.0 的 189CloudPC `family + stream` 场景曾在连续上传后出现 `initMultiUpload` 返回空 `uploadFileId`，WebDAV 层对外映射为 405；本轮通过PUT节流和短退避缓解，未修改 AList 驱动本身。
- 内置Compose固定为AList v3.61.0，但本轮不操作服务器；真实189CloudPC的PUT、MOVE、列出、下载和删除仍需部署后单独验证。
