本文只记录 `dev` 分支相对 `main` 的当前测试内容。确认稳定并合并到 `main` 时，把需要公开给用户的内容整理进 `CHANGELOG.md`，然后重置本文件。

## 当前状态

- 分支：`dev`
- 基准版本：`2.3.0`
- 当前 dev 提交：暂无已提交 dev 变更
- 镜像预期：推送 `dev` 分支后发布 `minori0721/bili-favorites-backup:dev`；`latest` 仍只由 `main` 分支发布。
- 本轮定位：本地缓存软上限，避免无人值守时下载持续占满本地 `temp`。

## 当前 dev 变更

### 本地缓存软上限

- 新增配置 `localCacheLimitGB`，默认 `10`，`0` 表示不限制。
- 统计范围只包括本项目 app 侧 `temp` 目录，不统计 `data`、AList 的 `alist` 数据目录或远端网盘空间。
- 当 `temp` 用量达到软上限时，下载队列不会启动新的下载任务；任务仍保持在“待下载”，不会进入失败、重试或消耗重试次数。
- 上传队列不受缓存上限影响，上传完成后的本地目录清理会刷新缓存状态并唤醒下载队列。
- 画质重调下载阶段复用下载队列，也受缓存软上限限制；上传替换完成并清理本地目录后会刷新缓存状态。
- 队列看板会显示本地缓存用量；超限时提示“下载暂停”，并说明上传队列仍继续运行。

## 测试重点

- 默认配置加载后应出现 `localCacheLimitGB: 10`；保存配置时允许 `0` 到 `1024`。
- `temp` 用量低于上限时，普通下载和画质重调下载能正常进入“下载中”。
- `temp` 用量达到或超过上限时，新下载任务保持“待下载”，不应标记失败、不进入 `retry_wait`、不增加重试次数。
- 缓存超限时，上传任务仍能进入“上传中”并继续清理本地目录。
- 上传清理、画质重调清理或“清理数据 -> 临时文件”后，缓存状态刷新，低于上限时下载队列自动继续。
- 队列看板在缓存开启时显示用量，超限时显示下载暂停提示。

## 建议测试命令

```bash
npm ci
npm run build
npm --prefix . audit --omit=dev
```

> 注意：`npm audit --omit=dev` 可能会报告依赖传递风险；自动修复前需要确认不会降级或破坏主依赖。

## 已知问题 / 暂缓项

- 本地缓存上限是软上限：如果开始下载时低于上限，单个视频下载完成后仍可能让 `temp` 超过上限；超过后才会暂停后续新下载。
- 本机不跑 Docker 构建，真实容器和 dev 镜像发布后仍需在服务器验证。

1. 在 `dev` 环境完成本轮变更对应的测试路径。

3. 确认 README 只保留稳定用户需要看的内容，dev 测试细节继续留在本文件。
4. 合并 `dev` 到 `main` 后，确认 GitHub Actions 只为 `main` 发布 `latest`。
5. 如果要保留下一轮 dev 测试，把本文件重置为新的 dev 状态。
